# ぷよぷよ × 強化学習オーケストレーション 設計ドキュメント

> 現行の固定探索 worker + PPO manager 実装は `docs/development/puyo-strategy-orchestration.md` を正本とする。この文書の学習 worker や高度な option 構成は将来案として扱う。

> 目的: 探索アルゴリズムや戦略のハードコーディングではなく、**戦略の選択そのものと連鎖構築の最善手選択を、強化学習で自動的に学習させる**ぷよぷよ AI をゼロから構築する。階層型強化学習(Hierarchical RL)の「戦略マネージャ × 特化ワーカー」というオーケストレーション構造を中核に据え、自己対戦・人間対戦を通じて全体を洗練させる。
>
> 想定読者: 自分自身。Claude Code / Codex に本ドキュメントを仕様として渡し、モジュール単位で実装を進めるための設計書。

---

## 0. 前提と設計判断（最初に握る）

このドキュメントでは以下を初期の前提として進める。いずれも後から変更可能だが、最初に固定しておくと迷いが減る。

- **対象ルール**: ぷよぷよ通(Tsu)ルールを基準とする。4 色、6×12(+ 隠し 13 段目)、3 列目をスポーン/窒息列とする。Fever 等の特殊ルールは対象外。
- **最終ゴールは対戦(versus)**。ただし最初から対戦を作らず、まず 1 人用(とことん連鎖)で学習パイプラインを成立させる。
- **言語は Python ファースト**。シミュレータの正確性を最優先で作り、速度が問題になったら該当部分のみ Rust(PyO3/maturin)か、JAX による完全ベクトル化に置き換える。
- **行動空間は「設置位置(列 × 回転)」の離散約 22 通り**。フレーム単位の低レベル操作では学習させない（理由は §3 参照）。
- **アルゴリズムは PPO を主軸**にし、必要に応じて AlphaZero 系(MCTS+NN)を比較対象として検討する。

> **最重要原則**: このプロジェクトの成否はシミュレータの正確性と速度でほぼ決まる。連鎖判定・スコア計算・おじゃま計算が 1 箇所でも間違うと、その上の強化学習は「間違った世界」を最適化し、原因切り分けが極めて困難になる。**シミュレータは強化学習を一切知らない純粋ロジックとして作り、ゴールデンテストで固定する。**

## 1. システム全体アーキテクチャ

4 つのレイヤに明確に分離する。下位レイヤは上位を知らない（依存は一方向）。

```
┌──────────────────────────────────────────────────────────┐
│ ④ 学習・評価基盤  (self-play league / rating / 人間対戦UI)  │
├──────────────────────────────────────────────────────────┤
│ ③ エージェント層  (マネージャ方策 + 特化ワーカー方策群)       │
├──────────────────────────────────────────────────────────┤
│ ② RL環境層  (観測エンコード / 設置位置行動 / 報酬設計)        │
│            Gymnasium(1人用) / PettingZoo(対戦) 準拠         │
├──────────────────────────────────────────────────────────┤
│ ① ぷよぷよシミュレータ  (盤面・連鎖・スコア・おじゃま・ツモ)   │
│            強化学習に非依存・単体でテスト可能                  │
└──────────────────────────────────────────────────────────┘
```

このレイヤ分離が後で効いてくる。たとえばシミュレータの正しさはエージェントと無関係に検証でき、行動空間の設計変更はシミュレータに波及しない。Claude Code / Codex に作業を委ねる際も「このモジュールだけ」と範囲を切れるので品質管理がしやすい。

## 2. ① ぷよぷよシミュレータ層 (`puyo_core`)

強化学習を一切 import しない、純粋なゲームエンジン。ここが全ての土台。

### 2.1 盤面表現
- フィールドは 6 列 × 12 段(可視) + 1 段の隠し段(計 13 段)。`numpy.int8` の 2 次元配列で「空 / 4 色 / おじゃま」を表す。
- 後でベクトル化(複数ゲーム同時シミュレート)するため、最初から「1 つの盤面 = 配列のスライス」として扱える設計にしておくと移行が楽。
- 窒息判定: スポーン列(3 列目、0-indexed で 2)の可視最上段が埋まり、新しい組ぷよを置けなくなったらゲームオーバー。

### 2.2 組ぷよ・操作
- 操作中の組ぷよは 2 個 1 組(軸ぷよ + 子ぷよ)。
- 回転は 4 状態(子ぷよが軸の上/右/下/左)。壁際での回転補正(壁蹴り)の挙動はルールにより異なるため、リファレンス実装に合わせて明示的にテストする。

### 2.3 連鎖判定・重力
- 消去ルール: 同色が 4 個以上、上下左右に連結したグループを同時消去。複数グループは同ステップで同時に消える。
- 重力: 消去後に浮いたぷよを落下させ、再度 4 連結ができたら次の連鎖として消去。これを連鎖が続く限り繰り返す。
- おじゃまぷよは色では消えないが、隣接する色ぷよが消えると巻き込まれて消える。

### 2.4 スコア計算（正確性が最重要）
1 連鎖ステップのスコアは概ね次の構造。

```
ステップスコア = (そのステップで消えたぷよ数 × 10) × 倍率
倍率 = clamp( 連鎖ボーナス + 連結ボーナス + 色数ボーナス, 最小1 )
```

- **連鎖ボーナス**(連鎖が深いほど増加)、**連結ボーナス**(1 グループの大きさ)、**色数ボーナス**(同時に消えた色数)の 3 種類の加算で倍率が決まる。
- **これらの厳密な数値テーブルは記憶で実装せず、Puyo Nexus 等の公開リファレンスから正確に転記すること。** 1 つでも値がずれると AI は「実在しない最適化」を学ぶ。転記したらゴールデンテスト(下記)で永久に固定する。

### 2.5 おじゃま計算・相殺
- 送るおじゃま数 = `floor(累積スコア / 目標点)`。目標点は通ルール既定で 70。端数は次回に繰り越す。
- 相殺(そうさい): 自分が送るおじゃまと、相手から降ってくる予定のおじゃまを打ち消し合う。落下前のおじゃまは「予約」状態として管理し、相殺ロジックを通す。
- 落下: 1 度に降るおじゃまの上限(例: 5 段 = 30 個)などの制約もリファレンスに合わせる。

### 2.6 ツモ(ぷよ列)生成
- 乱数シードで完全再現可能にする。`PuyoSequence(seed)` のように。
- 通ルールでは色の出現に制約(序盤は 3 色など)があるため、リファレンスに合わせる。
- **自己対戦では両プレイヤーに同一ツモを与える**(運要素を排し、純粋に手の優劣を比較するため)。これは公平な対戦と再現性に必須。

### 2.7 ゴールデンテスト（このプロジェクトで最も価値あるテスト）
有名な定形・連鎖の「初期盤面 → 期待されるスコア/連鎖数/送りおじゃま数」を多数用意し、シミュレータの出力が一致することを検証する。

- 1 連鎖、2 連鎖、3 連鎖の最小例
- 階段積み・カギ積み・GTR などの有名定形を組んで発火したときの連鎖数とスコア
- 同時消し、複数色同時消し、おじゃま巻き込み
- 相殺が成立/不成立になる境界ケース

このテスト群が緑なら、以降どれだけ強化学習がおかしな挙動をしても「環境は正しい」と断言できる。**ここに最初の数日を全力投資する価値がある。**

## 3. ② RL 環境層 (`puyo_env`)

シミュレータを強化学習から見えるインタフェースに包む。1 人用は Gymnasium、対戦は PettingZoo に準拠させると既存ライブラリと噛み合う。

### 3.1 観測(observation)
ニューラルネットに渡すテンソル。最低限：
- **自盤面**: 6×13 を色ごとに one-hot した `(色数+おじゃま, 13, 6)` のチャンネルテンソル（CNN で処理）。
- **ネクスト/ネクネク**: 見えている 3 組分の組ぷよの色情報。
- **おじゃま情報**: 自分に降る予定の量、相手に送る予約量。
- **(対戦時) 相手盤面**: 同様にエンコード。「相手の凝視」を学習させる入口になる。

### 3.2 行動空間（設計上の最重要判断）
**設置位置レベルの離散行動を採用する。** 「最終的にどの列に、どの回転で置くか」を選ぶ。6 列 × 4 回転から不正配置を除いて約 22 通り。

- 利点: 探索空間が桁違いに小さく、学習が圧倒的に速い。代表的なぷよ AI が設置単位で探索するのと同じ抽象度。
- 低レベル操作(左右移動・回転・落下のフレーム入力)は、選ばれた設置位置へ到達する経路探索(`actions.py` の pathfinding)で機械的に生成する。学習対象にしない。
- 不正な設置は行動マスク(action masking)でそもそも選べないようにする。これだけで学習効率が大きく改善する。

### 3.3 報酬設計
報酬ハッキングの温床なので慎重に。
- **1 人用(とことん)**: 連鎖で得たスコア(あるいは送りおじゃま量)を報酬に。窒息で大きな負の報酬。まずはこれでパイプラインを検証。
- **対戦**: 本質的な報酬は**勝敗(+1 / -1)のみ(疎報酬)**。ただし疎すぎて学習が進まないので、補助的な整形報酬(送ったおじゃま、組めた連鎖規模、生存ターン)を小さく足す。整形報酬は「勝つこと」と相関する範囲に留め、過剰にすると AI がそればかり最適化して歪む。
- 整形報酬は段階的に減衰(curriculum)させ、最終的に勝敗主体へ寄せていくと安定しやすい。

## 4. ③ エージェント層 — オーケストレーションの中核

ここが今回やりたいことの本体。**まずフラットな単体エージェントで土台を固め、その後で階層化する。**

### 4.1 ネットワーク基盤 (`networks.py`)
- 盤面を CNN(残差ブロック数層程度)でエンコードし、ネクスト等のベクトル特徴と結合。
- そこから方策ヘッド(行動確率)と価値ヘッド(状態価値)を出す Actor-Critic 構成。
- マネージャ/ワーカーで CNN バックボーンを共有するか分離するかは設計選択(§4.3)。

### 4.2 フェーズ1: フラットなベースライン (`flat_ppo.py`)
- 階層なしの単一 PPO エージェント。1 人用で「スコアが伸びる学習曲線」が出ることを確認する。
- **これはオーケストレーションの比較対象(ベースライン)としても必須。** 階層化して本当に強くなったのかを測る基準線になる。

### 4.3 フェーズ2以降: 階層型 = オーケストレーション

「戦略をマネジメントするモデル」＝**マネージャ方策**、「各戦略に特化したサブモデル」＝**ワーカー方策**として実装する。これは LLM のオーケストレーション(ルータ/プランナが専門サブエージェントに委譲する)と構造的に同型。

- **マネージャ方策 (`manager.py`)**: 盤面・相手盤面・おじゃま状況など全体を観測し、**どのワーカーに制御を委譲するか**を選ぶ。粗い時間粒度で意思決定する（毎手ではなく、連鎖発火・おじゃま着弾・N 手ごと等の「局面の節目」で戦略を切り替える）。マネージャの報酬は長期の勝敗。
- **ワーカー方策 (`workers.py`)**: それぞれが特定戦略に特化したサブ方策。設置位置を出力する。各ワーカーは「戦略固有の整形報酬 + マネージャから委譲された目的」を最大化するよう学習する。
  - 例: 大連鎖構築 / 速攻・潰し / 対応・カウンター / 凝視ベース防御。

階層化には複数の実装方針がある。**おすすめは(A)から始め、慣れたら(B)(C)へ。**

**(A) 明示的な戦略スロット + 学習ゲーティング(推奨スタート)**
- 戦略の種類を人間が定義(大連鎖/速攻/対応…)。各ワーカーに戦略固有の整形報酬とカリキュラムを与えて特化させる(例: 大連鎖ワーカーは単発の大連鎖に報酬、速攻ワーカーは早期のおじゃま送出に報酬)。
- マネージャは「どのワーカーを呼ぶか」を真の勝敗報酬で学習。
- 利点: **解釈可能・デバッグ可能・オーケストレーションの意味が明確**。学習用途・業務還元用途として最初に作るべき形。

**(B) オプション・フレームワーク**
- 各ワーカーを「オプション」(独自方策 + 終了関数)として定義。マネージャがオプションを選び、終了条件まで実行。Option-Critic / FeUdal Networks の系譜。
- 利点: 戦略の切り替えタイミングまで学習で決まる。より自律的。

**(C) 創発的なスキル分化**
- 戦略を人間が定義せず、相互情報量最大化(DIAYN 等)や情報ボトルネックで**ワーカーが勝手に多様な戦略に分化する**よう仕向ける。
- 利点: 「戦略そのものを探索的に学習」という当初の理想に最も近い。難易度は高いので、(A)(B) で基盤が固まってから挑む研究的フェーズ。

> 「相手盤面を凝視して戦略を切り替える」という ama 的な振る舞いは、(A)〜(C) いずれでも**マネージャの観測に相手盤面を入れる**ことで自然に学習対象になる。ハードコードした凝視ロジックではなく、「相手の状態 → 戦略選択」の写像を強化学習で獲得させる、という構図。

## 5. ④ 自己対戦・学習基盤

### 5.1 自己対戦とリーグ (`selfplay/`)
- 最新エージェント同士をぶつけるだけだと、ジャンケン的に戦略が循環したり、過去に勝てた相手に勝てなくなる(破滅的忘却)。
- **対戦相手プール**に過去の自分のスナップショットを蓄積し、確率的に過去版とも対戦させる(AlphaStar のリーグ学習の考え方)。
- **Elo レーティング**で強さの推移を定量化。学習が本当に進んでいるかの主指標。

### 5.2 人間対戦 (`eval/human_play.py`)
- 人間がブラウザ等で AI と対戦できる UI。学習済みエージェントを相手プールに混ぜたり、人間プレイをデモンストレーションとして収集したりできる。
- 最初は CLI/簡易描画で十分。後で `viz/renderer.py`(pygame か Web)に発展させる。

## 6. 学習基盤・横断的関心事 (`train/`, 共通)

- **並列化/スループット**: Puyo は状態空間が広く、数千万〜数億ステップが必要になりうる。Python シミュレータが遅ければベクトル化環境(複数ゲーム同時)か Rust 化で対処。最初から「環境はバッチで回す前提」で書いておく。
- **設定管理**: Hydra 等でハイパラ・実験設定を YAML 管理し、再現可能にする。
- **ロギング**: TensorBoard / Weights & Biases で学習曲線・Elo・各ワーカーの使用頻度を可視化。
- **チェックポイント/シード固定**: 完全再現できること。リーグのスナップショットもここから。

## 7. 技術スタック選定

| 領域 | 推奨 | 理由 |
|---|---|---|
| 言語 | Python(ファースト) | エコシステムが最も厚い。ボトルネック部のみ Rust(PyO3/maturin)へ。 |
| DL | PyTorch | RL 実装の参照が豊富。 |
| 環境 API | Gymnasium(1人用) / PettingZoo(対戦) | 既存ライブラリと噛み合う標準。 |
| RL 実装 | **CleanRL を起点に拡張** | 単一ファイルで全実装が読める。「内部を理解して業務還元する」目的に最適。階層型は自前拡張になる。 |
| 高速検証 | Stable-Baselines3(任意) | PPO/DQN をすぐ試して土台検証だけ済ませる用途。 |
| 高スループット | SampleFactory / PufferLib(任意) | 大規模学習が必要になったら検討。 |
| 探索系比較 | AlphaZero 系自前実装(任意) | MCTS+NN を RL と比較したくなったら。 |

> 理解と業務還元が目的なら、ブラックボックスな高機能フレームワークより **CleanRL のように内部が全部見える実装を読み・改造する**方が学びが大きい。階層型 RL は既製の決定版ライブラリが無いため、いずれにせよ自前構築になる点も CleanRL 起点と相性が良い。
>
> ライブラリのバージョンや最新動向は変化が速いので、着手時点で各公式ドキュメントの現行版を確認すること。

## 9. 開発ロードマップ（Claude Code / Codex 駆動前提）

各フェーズで「完了の定義(DoD)」を満たすまで次へ進まないこと。

**フェーズ0 — シミュレータ + テスト（土台）**
- `puyo_core` をテスト駆動で実装。連鎖・スコア・おじゃま・ツモ。
- DoD: ゴールデンテスト(§2.7)が全て緑。複数局を高速にシミュレートできる。

**フェーズ1 — 1人用環境 + フラット学習（パイプライン検証）**
- `single_env` + `flat_ppo` で「とことん連鎖」を学習。
- DoD: 学習曲線(平均スコア)が明確に右肩上がり。これでシミュレータ・環境・学習ループの結合が正しいと確認できる。

**フェーズ2 — 対戦環境 + 自己対戦（フラットのまま）**
- `versus_env` + 相手プール + Elo。
- DoD: 自己対戦で Elo が上昇。ランダム/単純ヒューリスティック相手に高勝率。

**フェーズ3 — 階層化（オーケストレーション導入）**
- §4.3(A) の「明示的戦略スロット + 学習ゲーティング」を実装。マネージャ + 複数ワーカー。
- DoD: 階層型がフラットなベースラインに Elo で有意に勝ち越す。マネージャが局面に応じてワーカーを使い分けている(使用頻度ログで確認)。

**フェーズ4 — 高度化**
- 相手盤面をマネージャ観測に入れて「凝視 → 戦略切替」を学習。オプション化(B)や創発的スキル分化(C)。人間対戦の組み込み。

**フェーズ5 — 業務還元**
- 学びを言語化・一般化。オーケストレーション・パターンを社内向けに抽象化(§11)。

## 10. 落とし穴・注意点

- **シミュレータの誤りが全てを汚染する**。スコア/おじゃまの数値テーブルは記憶で書かず、リファレンスから転記してゴールデンテストで固定。
- **行動空間を低レベルにしない**。設置位置レベル + 行動マスクで学習が一気に楽になる。
- **報酬ハッキング**。整形報酬が強すぎると本来の目的(勝つこと)から逸れる。疎報酬主体に寄せる。
- **自己対戦の不安定性**。最新版同士だけだと戦略が循環/忘却する。過去スナップショットのプールで防ぐ。
- **HRL のクレジット割り当ては難しい**。マネージャの行動(戦略選択)が成否に効いたのかの帰属が曖昧になりがち。粗い時間粒度・明示的戦略から始めて切り分けを容易に。
- **スループット**。Python シミュレータは遅くなりがち。最初からバッチ実行前提で書き、ボトルネックを計測してから Rust/ベクトル化。
- **ツモの公平性**。対戦は両者同一ツモ・固定シードで。
- **まず動くものを最短で**。フェーズ1 の学習曲線が立ち上がる瞬間が最大のマイルストーン。そこまでは最小実装で突き進む。

## 11. 業務還元の観点（なぜこの題材が効くか）

今回作る「マネージャがルータとして専門ワーカーに制御を委譲する」構造は、**LLM エージェントのオーケストレーション(プランナ/ルータ → 専門サブエージェント・ツール)と構造的に同型**。違いは、方策がプロンプトで与えられるのではなく強化学習で獲得される点。

この体験から社内に持ち帰れる抽象パターン:
- タスクをサブ問題に分解し、専門特化したコンポーネントに委譲する設計の勘所
- 「いつ・どのコンポーネントに委譲するか」という上位意思決定(ルーティング/ゲーティング)の難しさと評価方法
- 解釈可能な明示的分解(A) から創発的分解(C) へのスペクトラム、その採用判断
- リーグ/相手プールに相当する「評価の多様性確保」「忘却回避」の考え方は、エージェント評価やモデル更新の文脈にも転用可能

まず手を動かして「効くオーケストレーション/効かないオーケストレーション」の肌感覚を得ることが、最大の還元価値になる。

## 付録 A. Claude Code / Codex の使い方（このプロジェクト向け）

- **本ドキュメントを仕様として渡し、モジュール単位で依頼する**。「`puyo_core/rules.py` の連鎖判定だけ、§2.3 の仕様で。テストも一緒に」のように範囲を切る。
- **テストファースト**。特にシミュレータは「期待スコアを書いたゴールデンテストを先に作って → 通す実装を書かせる」流れが相性抜群。AI が勝手に仕様を歪めても、テストが番人になる。
- **レイヤ依存方向を守らせる**(下位は上位を import しない)。レビュー時に必ず確認。
- **1 フェーズ = 1 つの動く成果物**。フェーズの DoD を満たしてから次へ。中途半端に全レイヤを並行で書かせない。
- **数値テーブルや外部仕様は AI の記憶に頼らせない**。リファレンス値を自分で確認して与えるか、テストで固定する。
