manager_rule のモデル化解析

現在のぷよぷよ AI における manager_rule は、盤面と対戦状況から戦術をルールで選び、 選ばれた worker がビームサーチまたは目的関数つき探索で実際の設置手を決める、解釈可能な router baseline です。

結論

INPUT

select_action(observation, info) が受け取る観測です。 observation には自盤面・NEXT、info には自他の simulator、 予告おじゃま、着弾期限、得点、合法手 mask などが入ります。

判断技術

強化学習ではなく、TacticalContext を作る短期 roll-out 予測と if/elif ルールで戦術を選びます。構築系 worker は Ama 風ビームサーチ、 対応系 worker は目的関数つき bounded search を使います。

OUTPUT

返す主出力は離散設置 action です。副産物として profile 名、選択理由、 予測連鎖・得点・攻撃、危険度、探索ノード数、N手 plan などの diagnostics を保持します。

重要: manager_rule そのものは学習済みニューラルネットではありません。 PPO で学習する manager と同じ worker 実行部を使う一方で、worker/profile の選択を固定ルールで行う比較対象です。

実行経路

1. policy 生成 make_policy("manager_rule") が RuleBasedManagerPolicy を返す。
→
2. 戦術文脈化 build_tactical_context(info) が攻撃予測・危険度・期限を計算する。
→
3. profile 選択 recommended_strategy を profile ID に対応づける。
→
4. worker 探索 StrategyOrchestrator.propose() が worker を実行し設置 action を返す。

主要コード

ファイル 役割
selfplay/policies.py manager_rule という policy 名を RuleBasedManagerPolicy に解決する。
agents/strategy_manager.py ルールベース manager と、PPO checkpoint を読む学習済み manager の inference を定義する。

INPUT の内訳

observation

  • board: 自分の盤面テンソル。学習済み manager では CNN 入力にもなる。
  • next_pairs: current と NEXT を含む可視ツモ情報。
  • scalars や追加特徴: flat/checkpoint policy 用の補助情報。manager_rule の中心判断は主に info 側。

info

  • simulator: 自分側の HeadlessPuyoSimulator。合法手展開と未来手 preview に使う。
  • opponent_simulator: 相手側 simulator。相手の即時攻撃力・危険度・容量を推定する。
  • incoming_ojama / pending_ojama: 自分に来る予告おじゃま。
  • incoming_turns / incoming_ticks: 着弾までの猶予。
  • opponent_pending_ojama: 相手側の予告おじゃま。相手の窒息余地計算に使う。
  • score / opponent_score / step_count: 診断や学習 manager の特徴に入る。
  • action_mask: 呼び出し元 policy によっては合法 action の mask として使われる。

simulator がある場合、AI は盤面を直接いじるのではなく clone した headless simulator に action を適用し、 連鎖数・得点差・ゲームオーバー・盤面危険度を評価します。これにより UI に依存しない探索ができます。

manager_rule が判断するもの

manager_rule が直接選ぶのは「設置位置」ではなく「どの戦術 profile を使うか」です。 その後、選ばれた worker が設置 action を探索します。

戦術 profile 目的 主な探索設定 使われる worker
build_large 安全な局面で大きな連鎖を構築する。 depth 6 / width 32 / minimum chain 6 ビームサーチ worker
build_budget 計算量を抑えつつ連鎖を作る。 depth 3 / width 16 / minimum chain 4 ビームサーチ worker
punish 相手を倒せる推定攻撃量が即時にあるとき、致死量を狙う。 depth 3 / width 18 目的関数つき戦術 worker
counter 来るおじゃまを期限内に相殺できるとき、必要攻撃量を返す。 depth 3 / width 18 / safety margin 2 目的関数つき戦術 worker
fire_max 盤面が危なく、即時に撃てる攻撃を吐き出す。 1手探索 目的関数つき戦術 worker
survival 窒息回避と盤面安定を優先する。 1手探索 / danger tolerance 0.55 目的関数つき戦術 worker

戦術選択ルール

  1. 自分と相手の短期攻撃 forecast、盤面危険度、予告おじゃま、着弾期限を推定する。
  2. 来る攻撃が危険で、期限内に返せるなら counter を選ぶ。
  3. 来る攻撃が危険で、期限内に返せないなら survival を選ぶ。
  4. 即時攻撃が相手の推定 lethal target に届くなら punish を選ぶ。
  5. 自盤面危険度が高いなら survival を選ぶ。
  6. 即時攻撃が大きく、構築継続の安全性が低いなら fire_max を選ぶ。
  7. 上記に当てはまらない通常局面では build_large を選ぶ。

ビームサーチを利用した部分

ビームサーチは manager_rule の戦術選択そのものではなく、 build_large と build_budget が実際の設置手を決める段階で使われます。 実装は BeamSearchPolicy を BeamStrategyWorker が profile 設定で包む構造です。

探索木

現在の合法手を root action として clone simulator に適用し、次手以降も合法手を展開します。 各 node は「root action」「clone 済み simulator」「評価値」「これまでの最大連鎖価値」「早すぎる発火 penalty」を持ちます。

枝刈り

各 depth で評価値の高い node だけを width 件残します。 同じ盤面 fingerprint の候補は、より価値が高いものだけ残して重複展開を抑えます。

未知ツモ

current と可視 NEXT は維持し、そこから先の未知ツモを代表シナリオで置換します。 scenario 数を増やすと未知ツモへの頑健性が増えますが、計算量もほぼ比例して増えます。

評価関数

連鎖数と得点を大きく評価し、短すぎる連鎖の早期発火を penalty にします。 静的盤面評価では 2連結・3連結・発火可能性を加点し、孤立ぷよ、凸凹、井戸、危険高さ、おじゃまを減点します。

ビームサーチの INPUT / OUTPUT

項目 内容
INPUT observation と info["simulator"]、 profile 由来の depth、width、scenarios、 minimum_chain_count、連鎖/得点/早期発火 penalty の重み。
判断 clone simulator で合法配置を展開し、連鎖成果と盤面形状評価を合成した値で候補を残す。 root action ごとに最良値を集計し、最大値の action を選ぶ。
OUTPUT 設置 action。diagnostics として elapsed seconds、expanded nodes、scenario count、 action 別 candidate value を保持する。worker はさらに preview して予測連鎖・得点・攻撃・危険度を SearchProposal に格納する。

目的関数つき戦術探索

punish、counter、fire_max、survival は、 純粋な連鎖構築ではなく局面目的を満たすための bounded search を使います。 objective_for_profile() が戦術ごとに target attack、target score、target chain、deadline、許容危険度を設定します。

objective 評価の中心
punish / counter target attack 到達を非常に大きく加点し、不足量、過剰攻撃、遅さ、危険度を減点する。 counter は incoming と deadline に合わせて期限を決める。
fire_max 即時攻撃量、連鎖数、得点を加点し、危険度を減点する。
survival 静的盤面評価と低危険度を強く重視し、攻撃量は小さく加点する。
build fallback 的に盤面評価、連鎖、危険度から値を作る。

探索結果は ObjectiveResult として、目標達成、期限内達成可能性、未達理由、 surplus attack、score/chain delta、danger excess まで診断できます。

OUTPUT の内訳

主出力

  • select_action() の返り値: 対戦環境に渡す離散 placement action。
  • action は puyo_env.actions.action_to_placement() で axis x と rotation に変換される。

診断出力

  • current_profile_name: 今回選んだ profile 名。
  • tactical_diagnostics: incoming、target、deadline、reason、objective、objective result。
  • plan_diagnostics: N手 plan、各手の配置セル、予測盤面、予測攻撃、再計画条件。
  • SearchProposal: 予測連鎖・得点・攻撃、危険度、探索時間、展開ノード数、候補価値。

GUI や arena CSV では、これらの診断を使って「なぜその profile を選んだか」「どれくらい探索したか」 「目標を達成したか」を外から確認できます。

強化学習を利用した部分

強化学習は manager_rule の内部では使われていません。 ただし、このリポジトリには同じ worker 群を使う学習済み manager があり、 manager_rule はその比較対象として重要です。

PPO manager のモデル化

項目 学習済み manager manager_rule
選ぶもの worker/profile または tactical option と search-control の組。 ルールで選んだ worker/profile。search-control は使わない baseline 実行。
入力 自盤面 board、可視ツモ next_pairs、 manager_features。features には incoming、deadline、攻撃 forecast、危険度、 lethal margin、counter deficit、直前 profile、探索時間などが入る。 主に info から作る TacticalContext。 ニューラルネット用の feature vector は作らない。
判断技術 PuyoActorCritic の actor が離散 manager action の logits を出し、 deterministic 推論では argmax、学習時は categorical sampling を行う。 if/elif で recommended strategy を決め、profile 名 alias で worker に接続する。
学習 ManagerSelfPlayEnv 上で PPO。reward は対戦 reward に加え、切替 penalty、 判断時間 penalty、search cost、戦術 auxiliary reward を含む。 学習しない。重みや threshold はコード上の固定値。
出力 manager action を worker/profile と search-control に decode し、 worker が最終 placement action を返す。 profile を固定ルールで決め、worker が最終 placement action を返す。

PPO 学習ループで何を最適化しているか

  1. ManagerSelfPlayEnv が「manager の action space = decision 数 × search-control 数」と定義する。
  2. 各 step で actor-critic が action、logprob、value を出す。
  3. 選ばれた action は profile/control に decode され、orchestrator が worker を実行して placement action を得る。
  4. 対戦環境を進め、勝敗・得点・攻撃・相殺などに基づく reward を受ける。
  5. 切替や探索時間を penalty にし、推奨戦術に沿った成功には auxiliary reward を足す。
  6. GAE で advantage を計算し、clipped PPO loss、value loss、entropy bonus でネットワークを更新する。

したがって「強化学習が直接ぷよの置き場所を選ぶ」のではありません。 強化学習は局面ごとの戦術 worker と探索予算を選び、実際の配置品質は worker のビームサーチ/戦術探索に依存します。

全体の入出力まとめ

段階 INPUT 技術/判断 OUTPUT
戦術文脈化 info の simulator、相手 simulator、incoming、deadline、pending、score。 clone rollout による短期攻撃 forecast、盤面高さとおじゃまから危険度推定、相手容量から lethal target 推定。 TacticalContext。
manager_rule routing TacticalContext。 危険な incoming、counter 可否、lethal 可否、自盤面危険度、即時発火価値を順に見る固定ルール。 profile ID と profile 名。
objective 化 profile と TacticalContext。 戦術別に target attack、deadline、target chain、許容 danger、fallback を設定。 TacticalObjective。
worker 探索 profile、objective、observation、info simulator。 構築系はビームサーチ。対応系は目的関数つき bounded search。 SearchProposal と最終 placement action。
対戦環境 placement action。 ぷよ配置、落下、連鎖解決、おじゃま送受信、ゲーム終了判定。 次の observation、reward、done、info、diagnostics。
