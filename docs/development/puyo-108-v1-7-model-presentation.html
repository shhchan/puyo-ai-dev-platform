PUYO-108 v1.7.x 新モデル設計プレゼン

合意確認用の説明資料。`manager_rule` の固定ルール router から，局面解析 API と学習型 Strategy Manager， parameterized beam worker を組み合わせた階層型モデルへ進める設計案を整理する。

目次

設計ゴール アーキテクチャ スキーマ Manager 評価ロジック データセット作成 学習方法 各ステップ I/O 評価・合意確認

1. 設計ゴール

v1.7.x の基本方針は「大連鎖志向だが，相手リスクに反応できる AI」を作ること。 現状の `manager_rule` は，局面に応じて固定 worker を選ぶ解釈可能 baseline だが，戦術選択・戦術の中身・探索条件が学習で変化しない。

残すもの

  • ビームサーチを具体手探索の中核として使う。
  • 戦術単位で診断できる構造を残す。
  • UI / replay で判断理由を追えるようにする。

変えるもの

  • 固定6 profile の if 文 routing から脱却する。
  • 戦術のパラメータを学習対象にする。
  • 相手の本線以外の攻撃オプションを明示的に解析する。

将来拡張

  • `TacticSpec` の分岐・統合・改良。
  • latent option 化。
  • value model による候補手列 ranking。
2ダブは「短期高火力のサブ連鎖」の一例として扱う。設計対象は 2ダブ専用対応ではなく， 相手が本線以外に持つ攻撃オプション全体への対応である。

3. スキーマ定義

固定戦術名に閉じないため，観測・攻撃オプション・戦術・plan を構造化して扱う。 初期は人間が理解できる schema から始め，将来的には embedding や parameter distribution を学習対象にする。

3.1 State Analyzer 出力

schema 役割 主なフィールド 使い道
AttackOption 相手/自分が持つ攻撃候補を表す kind, turns_to_fire, expected_attack, attack_per_turn, chain_count, uses_main_chain 本線以外の短期高火力，副砲，速攻，リーサル候補を比較する
MainChainForecast 本線の伸びしろと発火可能性を表す estimated_chain, estimated_attack, trigger_accessible, extension_potential 大連鎖構築を継続するか，発火するかを判断する材料
BoardRisk 窒息・発火点喪失・形崩壊の危険度 danger, choke_risk, trigger_loss_risk, center_height 生存・安全構築・発火判断の制約
ResponseFeasibility incoming や相手攻撃への対応可能性 incoming_attack, deadline, max_return, counter_margin 完全相殺，上回る返し，受け，カウンター準備の比較
OpponentVulnerability 相手のスキを表す no_trigger_score, choke_risk, low_response_score, ojama_window 催促・速攻・リーサル狙いが通りやすいかを見る

3.2 TacticSpec

戦術を固定名ではなく仕様として表す。`applicability` は候補に出しやすい条件であり，固定 if 文ではない。 

TacticSpec
  identity:
    id / name / version / optional human_label
  applicability:
    candidate_features or learned_embedding
  objective:
    target_chain / target_attack / deadline
    counter_margin / pressure_goal
  constraints:
    danger_tolerance / trigger_preservation
    max_latency / avoid_choke
  planner:
    beam_depth / beam_width
    evaluation_weights / candidate_count
  termination:
    objective_achieved / timeout / danger_threshold / opponent_fired
  fallback:
    fallback_tactic / safety_behavior
  diagnostics:
    success_rate / mean_attack / mean_chain
    response_success / usage_clusters

初期 TacticSpec 例

build_main prepare_response counter_or_return pressure lethal_attack all_clear fire_main survive

将来の進化

  • 類似 tactic のクラスタリング。
  • 使用局面ごとの成功率で分岐・統合。
  • parameter distribution の学習。
  • latent option への置き換え。

3.3 Worker 入出力

項目 入力 出力
Parameterized Beam Worker 盤面，NEXT，`TacticSpec`，`target_attack`，`target_chain`，`deadline`，`danger_tolerance`，探索予算 1手目 action，N手 plan，予測連鎖，予測おじゃま，危険度，期限達成可否，発火点維持診断
Planner Preview 上位 tactic 候補，短い beam budget，現在局面 候補 tactic ごとの簡易 plan outcome と risk/cost

4. Learned Strategy Manager

初期実装は完全に独立した複数 sub agent ではなく，単一 manager の中に戦術別 proposal/evaluator head を持つ構造にする。 これにより，sub agent 的な解釈性を残しつつ，学習・同期・credit assignment を単純にする。

構造

  1. shared encoder が盤面 tensor と analyzer diagnostics を読む。
  2. 各 tactic head が「自分を使うならこの条件」という proposal を出す。
  3. 上位候補だけ worker で短く preview する。
  4. final arbitration head が tactic と parameters を決める。

評価値の考え方

固定式ではなく学習済み value / policy head が出す。

value(tactic) =
  learned_value(
    analyzer_features,
    tactic_embedding,
    planner_preview,
    previous_plan_state
  )

診断として保存する分解値

learned_value predicted_attack deadline_success danger_risk trigger_preservation latency_cost counter_margin
人間が重みを固定するのではなく，重み付けは学習対象にする。ただし，UI / replay では評価の内訳を見えるようにする。

5. データセット作成方法

Stage 1 の dataset は「初期対戦相手」ではなく，局面解析 API と学習/評価の足場である。 固定盤面・固定 NEXT・固定おじゃま状態を作り，State Analyzer が何を見ているかを確認できるようにする。

5.1 Scenario Dataset

scenario class 作る局面 期待する analyzer 出力 学習/評価での用途
短期高火力サブ連鎖 相手が本線以外に2〜3手以内の副砲を持つ `AttackOption(kind=sub_chain, attack_per_turn=high)` 攻撃オプション候補の検出と response 準備評価
本線保持 相手が大きく伸ばせる本線と発火点を持つ `MainChainForecast(estimated_chain, trigger_accessible)` 催促か構築継続かの判断材料
incoming 対応 自分におじゃまが落ちる期限付き局面 `ResponseFeasibility(max_return, deadline, counter_margin)` 完全相殺，上回る返し，生存優先の比較
相手のスキ 相手の発火点がない，受けが狭い，中央が高い `OpponentVulnerability(score=high)` 催促・速攻・リーサル狙いの評価
自滅リスク 自分の発火点喪失や窒息が近い局面 `BoardRisk(choke_risk, trigger_loss_risk)` survive / fire_main / safe build の評価
全消し 全消し直後，または全消しが狙える局面 `context(all_clear=true)` 専用 tactic と通常構築の切替評価

5.2 Dataset 作成フロー

  1. 代表局面を fixture として保存する。盤面，NEXT，おじゃま queue，得点，tick を含める。
  2. State Analyzer を実行し，structured diagnostics を保存する。
  3. sanity check を付ける。例: 「相手サブ連鎖候補が1件以上」「incoming deadline が2」など。
  4. 必要なら beam preview を実行し，各 tactic の簡易 outcome を保存する。
  5. replay / HTML / JSON artifact で人間が確認できる形にする。
{
  "scenario_id": "short-subchain-threat-001",
  "inputs": {
    "own_board": "...",
    "opponent_board": "...",
    "visible_next": "...",
    "ojama_queue": {"own": 0, "opponent": 0},
    "tick": 120
  },
  "analyzer_expected": {
    "opponent_attack_options_min": 1,
    "has_short_tactical_attack": true,
    "opponent_vulnerability": "low_or_medium"
  },
  "artifacts": {
    "replay": "path/to/replay.json",
    "diagnostics": "path/to/analyzer.json"
  }
}

6. 学習方法

最初から self-play のみで始めると破綻しやすい。まず観測 API と初期 policy を作り， その後 mixed opponent training，self-play，latent option へ進める。

Stage 1

Analyzer and Scenario Dataset

学習前でも局面を観測・説明できる状態を作る。

Stage 2

Bootstrapped Manager

scripted scenario，強い beam，人間ログ，既存 checkpoint で初期 manager を作る。

Stage 3+

Training and Evolution

mixed opponent，self-play，latent tactic evolution へ進む。

学習段階の詳細

stage 目的 入力 出力 完了条件
1. Analyzer 観測 API と scenario dataset の確立 固定局面，NEXT，おじゃま queue，ルール情報 Analyzer schema，scenario JSON，diagnostics artifact 代表シナリオで analyzer の sanity check が通る
2. Bootstrap 破綻しにくい初期 manager の作成 Analyzer 出力，scripted teacher，beam preview，人間/既存モデルログ 初期 manager checkpoint，behavior cloning metrics 自滅率が低く，最低限の構築/相殺/発火ができる
3. Mixed Opponent 固定相手・過去世代・人間由来モデルへ適応 初期 manager，opponent pool，報酬設計 強化済み checkpoint，対戦 metrics `manager_rule` / beam / baseline に対して改善傾向が出る
4. Self-Play 自己対戦で戦術を強化 mixed training 後 checkpoint，過去世代 pool self-play checkpoint，世代別 benchmark 戦術崩壊や自滅増加なしに勝率・連鎖・対応が改善
5. Latent Evolution 戦術 schema の分岐・統合・改良 TacticSpec usage logs，success metrics，latent embeddings 更新された tactic registry，option diagnostics 新 tactic が既存 tactic より有効な局面を持つ

8. 評価・合意確認

定量評価

  • 平均最大連鎖数。段階目標は平均10連鎖級，将来理想は平均14連鎖級。
  • 短期高火力サブ連鎖・速攻への対応成功率。
  • incoming 相殺成功率，incoming を上回って返せた割合。
  • 催促で相手の発火を誘発できた回数。
  • 自滅率，窒息率，発火点喪失率。
  • `manager_rule`，beam，worker baseline，既存 checkpoint への勝率。

GUI / Replay QA

  • 安全局面で本線構築を継続する。
  • 相手の本線以外の攻撃オプションに反応する。
  • 相手にスキがあるとき催促・リーサル狙いを選べる。
  • incoming に対して相殺，返し，生存を選び分ける。
  • 全消し局面で専用方針を取る。
  • 発火点を潰して自滅しない。

合意確認チェックリスト

  • 2ダブ専用ではなく，本線以外の攻撃オプション全般を解析対象にする。
  • Analyzer は観測 API であり，行動ルールを固定しない。
  • manager は単一 model + 戦術別 heads から始め，将来 sub agent / MoE に拡張できる形にする。
  • 戦術は `TacticSpec` schema で表現し，将来の動的生成・進化に備える。
  • Stage 1 は対戦相手作成ではなく，観測 API と scenario dataset 作成である。
  • Stage 2 は self-play 前の初期 manager bootstrapping である。
Open Questions は現時点では保留。次回以降，初期戦術カテゴリ数，Analyzer の初期精度，reward 重み， value model ranking を v1.7.x 初期に含めるかを確認する。
