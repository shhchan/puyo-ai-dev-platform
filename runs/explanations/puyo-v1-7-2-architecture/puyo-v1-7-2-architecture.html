最初に結論

PUYO-130が開始不可なのは、PUYO-130のtrainerに不具合が見つかったからではありません。 そのtrainerはまだTo Doです。開始前提である「探索workerが、学習で選び直す価値のある安全な大連鎖候補を、 runtime予算内で安定して提示できること」をPUYO-170 capability gateが満たさなかったためです。

K-bestの中を評価専用selectorで選び直しても平均最大連鎖は2.933で、要求値10に届きません。 さらに30 gameのうち1 gameが40手前にgame-overし、proposal p95は917.12 msで registry上限60 msを約15.3倍超過しました。したがって「rankerを学習すれば解決する」とはまだ言えません。

  1. 状態を構造化する。 Analyzerが双方のdanger、main-chain forecast、attack options、incoming deadline、全消しbonus lifecycleを作ります。
  2. 適用不能なtacticをmaskする。 incomingがある場合はresponse guardが、相殺可能ならcounter_or_return、不可能ならsurviveへ責務を固定します。
  3. 軽量headで全tacticを評価する。 tactic logit、value、risk、objective/constraint/planner parameterを出します。

  6. 選ばれたworkerがK候補を生成する。 build_mainならdiverse beam、response/tacticalなら対応workerを使います。
  7. K候補をversioned batchへ正規化する。 mask、stable ID、13 ranker feature、telemetryが揃っています。
  8. 現時点ではslot 0を実行する。 learned candidate rankingはまだ接続契約だけで、optimizerとlong-run trainingはPUYO-130です。

3. Capability gateで何が失敗したか

capability gateはruntimeのrank-0をそのまま評価しただけではありません。K候補から「目標10連鎖まで発火を急がず、 安全性と到達可能性が高いもの」を選ぶ評価専用selectorで30 seedのpathを作りました。 それでも基準に届かなかったため、単なるslot-0選択ミスではなくcandidate-generator側の前提不足と判断できます。

4. 「どこに問題があったか」の正確な読み方

Primary problem

候補生成pathの長期品質が不足

現行registry設定はd3 / w24 / probe8 / K8です。K候補を評価専用selectorで選んでも平均2.933連鎖でした。 最大11連鎖を作れるseedはあるため能力が完全にゼロではありませんが、30 seedへ一般化していません。

reference root coverage 83.7%、path coverage 65.1%は、より高予算な探索が選ぶrootやpath prefixの一部を 現行探索が保持できていないことを示します。horizon、pruning、候補diversity、BuildPotentialの長期表現のいずれか、 または複合がまだ不足しています。

Second problem

runtime予算を大きく超過

capability proposalの観測p95は917.12 msで、build_main registry budget 60 msの約15.3倍です。 これはproposal projectionを含むwall-clock測定であり、12 worker並列実行時の負荷も含むため、 単一realtime tickの厳密な代替値ではありません。

ただしPUYO-170が採用した再現可能なgate metricでは明確なFAILです。 「より深く探索して連鎖を伸ばす」と「realtime budgetを守る」の両方を同時に満たす構成がまだありません。

Nuance

27件のcandidate-limited premature

小連鎖発火27件は、同じK内に静かなno-fire候補があったのにselectorが誤選択した avoidable ではなく、K内にその候補がなかった candidate-limited です。 一方でavoidable candidate gapは0なので、単純な「legalな静かなrootをKから落とした」ケースではありません。

これはそれ以前の手順で盤面を行き止まりにし、発火を避けにくい状態へ到達した可能性を含みます。 問題は1 decisionのrankだけでなく、複数手にまたがるtrajectory品質です。

Healthy area

脅威対応は今回のblockerではない

PUYO-158後の小攻撃、burst、全相殺、余剰返し、生存、本線再開は6/6 PASSです。 avoidable gap、forced game-over candidate gap、schema、artifact hash、独立再生digestもPASSしました。

したがって「モデル全体が動かない」のではありません。安全盤面で長く本線を育てる能力と実行コストが、 threat-responseより弱いという局所的な品質問題です。

pretraining selected-policyの別指標: migrated bootstrap checkpointを通常runtime policyとして測ると、平均最大連鎖4.133、premature fire 86、early game-over 0でした。 threat scenarioは6/6ですが、これはpost-training checkpointではないためpromotionは PENDING_POST_TRAININGです。capability結果とpromotion結果を混ぜてはいけません。

5. なぜ先にRLを走らせず、何が揃えば解除できるか

先に止める理由

  • learned rankerはK候補の順序を変えられますが、候補集合にない長期planを作れません。
  • 不十分なaction setでlong-runすると、勝率だけ改善し、小連鎖早打ちを残す可能性があります。
  • 60 msを大幅に超える探索を前提に学習すると、realtimeで使えない方策になります。
  • gateはgenerator defectをRLで隠すことと、長時間computeの浪費を防ぎます。

推奨される解除順

  1. runtime budget内で長期path coverageを改善する。
  2. 30 seed × 40手でcapabilityを再評価する。
  3. 30/30完走、平均最大連鎖10以上、gap 0、p95 60 ms以下、決定論を同時PASSする。
  4. その後にPUYO-130でcandidate ranker / critic / rewardをlong-run学習する。
  5. PUYO-131でstate class別reward・regret・tactic usageを診断する。
  6. PUYO-133で5固定training seed、10/0/0、threat 6/6、GUI/replay、lineageをpromotion判定する。
