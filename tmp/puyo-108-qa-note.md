manager_rule の正体
manager_rule は「学習済みニューラルネットモデル」ではなく、ルールベースの戦術 router です。
agents/strategy_manager.py:220 の RuleBasedManagerPolicy が、局面から recommended_strategy を取り、それに対応する worker
profile を1つ選びます。その後、agents/strategy_workers.py:1510 がその worker だけを実行して、実際の設置手を探索します。

選べる基本 profile は6つです。定義は agents/strategy_workers.py:437 です。

- build_large: 安全時に大きめの連鎖を作る
- build_budget: 軽めの探索で構築する
- punish: 相手を倒せる攻撃量を狙う
- counter: 来るおじゃまを相殺/返す
- fire_max: 今すぐ撃てる最大火力を撃つ
- survival: 窒息しない配置を優先する

ルール分岐は agents/strategy_workers.py:887 にあります。ざっくり優先順位はこうです。

1. 危険なおじゃまが来ていて返せるなら counter
2. 危険なおじゃまが来ていて返せないなら survival
3. 即時攻撃で相手を倒せそうなら punish
4. 自盤面がかなり危険なら survival
5. そこそこ火力があり盤面安全度が低いなら fire_max
6. それ以外は build_large

つまり、現状の manager_rule は「人間っぽく方針を持って戦うモデル」というより、「危険/リーサル/相殺のしきい値で6種類の探
索器を切り替える baseline」です。v1.7.0 の新モデルを設計するなら、まずここを「どんな思考単位にしたいか」から決めるのがよ
さそうです。

今回の回答から，v1.7.0 / v1.7.x の新モデルに求めている方向性は，現状の `manager_rule` の延長というより，
「固定ルールで6 profileを切り替える baseline」から「相手状況を理解しながら，戦術の選択・戦術の中身・探索の目的を学習で調整していく階層型モデル」へ進めることだと理解しました．

現状の `manager_rule` との差分は次のように整理できます．

- `manager_rule` は学習しない．新モデルは self-play / 強化学習で判断が変化していく必要がある．
- `manager_rule` の戦術は `build_large` / `counter` / `punish` などの固定カテゴリ．新モデルでは戦術候補や戦術パラメータが学習で洗練される必要がある．
- `manager_rule` は単純なしきい値で相手対応する．新モデルでは「相手が2ダブを構えている」「相手にスキがある」などの局面理解を使って，攻撃・催促・構築・防御を切り替えたい．
- 設置候補の生成・探索にはビームサーチを使ってよい．ただし，ビームサーチの目的関数や候補評価は学習で改善する余地がある．
- 最初の成果物は実装ではなく，仕様・設計ドキュメント．後続チケットで v1.7.x として段階実装する前提でよい．

この前提だと，To-Be 像の候補は次のようになります．

```text
盤面・NEXT・おじゃま・相手盤面
        |
        v
局面解析器
  - 自分の連鎖/火力見込み
  - 相手の連鎖/火力見込み
  - 2ダブ/速攻/本線/発火点/窒息リスク
  - incoming/deadline/counter可能性
        |
        v
学習型 Strategy Manager
  - 今の目的を選ぶ
  - 目的の強さ・期限・安全許容度を出す
  - 必要なら latent option / continuous parameter を出す
        |
        v
Parameterized Beam Planner
  - manager が出した目的関数で候補手列を探索
  - 候補手列を評価し，1手目を action として返す
        |
        v
対戦結果・局面報酬・相殺/催促/連鎖成功を学習信号にして更新
```

重要なのは，「戦術をハードコードしない」と「学習を安定させる」の間にトレードオフがある点です．
完全に自由な latent action だけにすると学習は難しくなります．一方で，現状の6 profile固定に寄せすぎると `manager_rule` と同じ不満が残ります．
そのため，最初の設計案としては「固定戦術名を増やす」のではなく，以下のような parameterized / latent option 型がよさそうです．

- manager は `build` / `pressure` / `counter` のような粗い intent を持ってもよいが，最終的な攻撃量・期限・安全許容度・探索重みは連続値で出す．
- option 名は人間向け診断として付けるが，実体は `target_attack`，`target_chain`，`deadline`，`danger_tolerance`，`harass_weight` などのパラメータにする．
- 相手の2ダブなどの認識は feature extractor / analyzer で出してよいが，「それにどう反応するか」は強化学習で学ぶ．
- ビームサーチは候補生成器として残し，候補の価値評価・目的関数重み・探索予算を学習対象にする．

3. ビームサーチ側は，v1.7.x でどの役割にしたいですか？

   - opponent pool を使って過去世代や固定 baseline と戦わせる

   「ハードコードせずに強くなる」を重視するなら self-play は必要ですが，最初から self-play だけだと学習が不安定になりやすいです．

2 回目回答から，v1.7.x の基本方針は次のように置けそうです．

- 最終理想は「A -> B」．つまり，最初は少数の明示的な戦術カテゴリと学習パラメータで始め，段階的に latent option / schema 化された戦術へ拡張する．
- 局面解析器はハードコードしてよい．ただし，解析結果に対する反応ルールは固定せず，モデルが判断する．
- beam search は候補生成・具体手探索の worker として使う．manager は選ばれた戦略/worker に「期限」「目標連鎖」「目標火力」「安全許容度」「探索予算」などの条件を渡す．
- 学習は imitation / teacher / 人間や別モデルとの対戦で立ち上げ，その後 self-play で強化する．
- 成功基準は，まず平均 10 連鎖級を段階目標にしつつ，将来的には 14 連鎖級を理想とする．相手対応，勝率，UI 観察上の納得感，自滅しないことも評価に入れる．

この整理をもとに，`docs/development/puyo-108-v1-7-model-design.md` に仕様・設計初稿を作成する．

#### 2ダブへの言及について

指摘どおり，設計書の初稿は「短時間で高いおじゃまを送る攻撃」の代表例として 2 ダブに寄せて書きすぎていました．
To-Be として扱うべき対象は 2 ダブそのものではなく，相手が本線以外に持っている攻撃オプション全体です．

整理すると，State Analyzer は相手盤面から次のような「攻撃オプション候補」を列挙・評価する API であるべきです．

- 本線候補: 相手が最終的に大きく伸ばしそうな最大連鎖．
- サブ連鎖/副砲候補: 本線とは別に，短い時間で撃てる攻撃．
- 即時発火候補: 1手以内に撃てる攻撃．
- 短期発火候補: 2〜3手程度で撃てる攻撃．
- 催促/速攻候補: 連鎖数は小さくても単位時間あたりの火力が高い攻撃．
- 対応困難候補: こちらの現在形から見ると相殺や受けが難しい攻撃．

2 ダブはこの中の「短期発火候補」かつ「単位時間あたり火力が高い副砲候補」の一例です．
設計書では `double-attack` という語を消し，`short-tactical-attack` / `sub-chain threat` のような一般化した表現に置き換えます．

#### Architecture の入力について

初稿の `board / next / ojama / opponent board` は省略しすぎでした．実際には少なくとも次を入力として考慮すべきです．

- 自分と相手の盤面．
- 自分と相手の current pair / visible NEXT．相手の NEXT が観測可能なら相手分も含める．
- 自分と相手の pending / incoming おじゃま，落下期限，おじゃま queue．
- 自分と相手の得点，送信済み/相殺済み/受信済みおじゃま統計．
- 全消し状態，全消しボーナスの有無．
- 現在 tick / turn / 残り入力時間 / policy decision deadline．
- 直近の行動履歴，発火履歴，連鎖中かどうか．
- 前回 manager が選んだ intent / tactical parameters / plan，およびその成否．
- ルール・環境メタデータ．例: 盤面サイズ，色数，最大可視 NEXT，対戦/リアルタイムモード．

入力はすべて neural network にそのまま入れる必要はありません．一部は State Analyzer が要約し，manager には feature vector / structured diagnostics として渡すのがよいです．

#### `double-attack`, `rush`, `context`, `opponent vulnerability` の意味

- `double-attack`: 初稿では 2 ダブに近い意味で書いていました．これは不適切なので，短期高火力のサブ連鎖全般へ一般化します．
- `rush`: 速攻です．相手が序盤または短い準備で攻撃を仕掛ける状態・方針を指していました．これも `short-term attack` の一種として整理します．
- `context`: 文脈です．例として全消し直後，incoming がある，発火点が高い，連鎖中，残り時間が短い，など，盤面単体だけでは判断しにくい状態を指します．
- `opponent vulnerability`: 相手のスキです．相手が発火点を持っていない，中央が高い，受けが狭い，短期火力がない，おじゃまが落ちる直前，など「攻撃・催促が通りやすい度合い」を指します．

#### 任意の戦術を記述する schema について

これは入れるべきです．初期実装では明示カテゴリを使うとしても，後から戦術自体を作成・進化させるなら，戦術を `name` ではなく schema で表現する必要があります．

設計上は `TacticSpec` のような構造を置くのがよさそうです．イメージは次の通りです．

```text
TacticSpec
  identity:
    id / name / version / optional human label
  applicability:
    どの局面で候補に出しやすいかを表す特徴条件または learned embedding
  objective:
    target_chain, target_attack, deadline, counter_margin, pressure_goal
  constraints:
    danger_tolerance, trigger_preservation, max_latency, avoid_choke
  planner:
    beam depth, width, evaluation weights, candidate count
  termination:
    objective achieved, timeout, danger threshold, opponent fired
  fallback:
    失敗時に移る tactic または safety behavior
  diagnostics:
    成功率, 平均火力, 平均連鎖, 対応成功率, 使用局面クラスタ
```

初期は人間が `build_main` などの `TacticSpec` を定義し，manager がそのパラメータを学習します．
次段階では，`TacticSpec` の embedding や parameter distribution を学習して，似た戦術の分岐・統合・改良ができる形にします．

#### manager 一人か sub agent 方式か

初期実装では「一人の manager + 戦術別 proposal/evaluator head」が最も現実的だと思います．
完全に独立した sub agent を複数作ると，学習・同期・credit assignment・診断がかなり難しくなります．

ただし，人間の理解としては sub agent 的に見える構造にできます．

```text
shared encoder
  -> build_main head
  -> response head
  -> pressure head
  -> fire_main head
  -> survive head
  -> final arbitration head
```

各 head は「自分の戦術を使うなら，この条件でこのくらい価値がある」という proposal を出し，final arbitration head が最終選択します．
実装上は単一 model なので，学習は一体で行えます．将来的に必要なら mixture-of-experts や本当の sub agent に分離できます．

#### 各戦術の評価値をどう算出するか

初期案では，戦術評価値は固定式ではなく学習済み network が出します．ただし，入力として worker / analyzer から得た診断値を使います．

候補としては二段階です．

1. 軽量評価:
   - State Analyzer の特徴量と各 `TacticSpec` の embedding を入力する．
   - 各戦術の logit / expected value / risk を network が出す．
   - ここで数個の候補戦術に絞る．

2. planner-preview 評価:
   - 上位候補だけ beam worker で短く preview する．
   - 予測連鎖，予測おじゃま，期限達成，危険度，発火点維持，候補数などを得る．
   - manager が `learned_value + preview_outcome - risk_penalty - latency_penalty` のような形を学習済み critic/value head で評価する．

この式は人間が重みを固定するものではなく，network の出力として学習します．
ただし，診断ログでは「火力は高いが危険度が高い」「相殺可能だが期限が厳しい」のように分解して見えるようにします．

#### Stage 1 / Stage 2 について

Stage 1 は初期対戦相手を作る段階ではありません．
これは「モデルが学習で使う観測 API と評価用シナリオを整える段階」です．

Stage 1 で作るもの:

- State Analyzer の入出力 schema．
- 相手攻撃オプション，本線見込み，短期火力，スキ，incoming，相殺可能性などの診断値．
- 固定盤面・固定 NEXT・固定おじゃま状態の scenario dataset．
- 各 scenario に対する analyzer 出力の期待値，または sanity check．
- GUI / replay / CUI で analyzer が何を見ているか確認できる artifact．

Stage 1 の目的は，「学習前でも局面を観測・説明できる状態」を作ることです．

Stage 2 は，その Analyzer と scenario dataset を使って，初期 manager を作る段階です．
ここでは初めて policy に近いものができます．ただし，強い self-play model ではなく，学習開始地点として破綻しにくい初期 policy です．

Stage 2 で使う teacher は，必ずしも `manager_rule` だけではありません．
強い beam，scripted scenario，人間ログ，既存 checkpoint，手作業で作った局面評価などを混ぜられます．
目的は「最初から自滅する」「まったく相殺しない」「本線を伸ばせない」といった状態を避け，Stage 3 以降の対戦学習に乗せることです．
