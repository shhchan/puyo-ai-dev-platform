# AGENTS 運用ルール（puyo_ai_dev_platform）

このリポジトリでは、Codex は以下を既定動作とする。

## 1. Jira 起票の既定動作

- 対象: 非自明な実装依頼（調査・実装・テストを伴う依頼）
- 既存チケットが指定されている場合は、まず Jira でチケット内容・ステータス・親エピック・関連チケットを確認する。
- 既存チケットがない非自明な依頼では、作業開始前に Jira を機能単位で分割して起票する。
- 起票時は、内容と緊急度を見て以下を判断する。
  - バックログに置くか、現在のスプリントに入れるか、次のスプリント候補にするか。
  - 既存エピックへ紐づけるか、新しい親管理チケットが必要か。
  - 依存関係・ブロッカー・検証単位を別チケット化すべきか。
- Jira UI で追跡しやすくするため、サブタスクは原則作成しない。分割が必要な場合は独立したタスクとして作成し、親管理チケットまたは関連チケットへ `Relates` で紐付ける。
- チケットは 1 件ごとに独立して着手・レビュー・検証できる粒度にする。
- 起票直後に着手するチケットのみ `進行中` へ遷移する。後続・未着手チケットは適切なバックログまたはスプリント状態に残す。

## 2. 実装時の Jira 更新

- 作業開始時は、対象チケットのステータスを `進行中` へ遷移する。
- Jira 操作では、まず `atlassian` MCP の `getAccessibleAtlassianResources` で接続確認する。
- 本リポジトリの Jira site は `https://shhchan.atlassian.net`、cloudId は `46424ed5-7d42-4bff-bc2a-da4c296f8b5b`。
- Rovo Search が 403 を返す場合でも、cloudId を指定した Jira issue API（取得・コメント・遷移・起票）を優先して試す。
- Jira コメントは「1作業セッションにつき1件/チケット」を原則とする。
- コメントには以下を含める。
  - 実施内容の要約
  - 変更ファイルまたは変更範囲
  - テスト結果
  - 残課題（あれば）
- 作業完了時は、対象チケットのステータスを `COMPLETE` へ遷移する。
- `完了` ステータスへの遷移は人間が行うため、Codex は実行しない。

## 3. Git / GitHub 運用

- 作業開始前に対象チケット用のブランチを切る。
- ブランチ名は原則 `PUYO-13/xxx-yyy-zzz` の形式にする。
  - 先頭は Jira チケット番号。
  - 後続は作業内容を表すケバブケース。
- 既存の未コミット変更がある場合は、ユーザー変更を混ぜないようにステージ対象を明示的に限定する。
- 作業単位ごとに適宜 commit する。
- コミットメッセージは英語で端的に書く。
- 作業がすべて完了したら PR を作成してレビュー依頼を出す。
- PR タイトルは `[PUYO-13] タイトル（日本語）` の形式にする。
- PR description は日本語で、以下のフォーマットを使う。

```markdown
## What

## Why

## QA

## References
```

## 4. Codex 作業品質

- 実装前に対象ファイル・既存テスト・関連ドキュメントを確認し、既存設計と重複しない方針を選ぶ。
- 変更は小さく保ち、要求に直接関係しないリファクタリングや整形を混ぜない。
- ユーザー変更がある場合は上書きせず、必要なら差分を確認してから作業範囲を限定する。
- テストは変更内容に見合う最小十分な範囲で実行し、実行できない場合は理由と代替確認を明記する。
- 完了報告では、変更内容・変更ファイル・テスト結果・残課題・人間が確認できる手順を簡潔に示す。

## 5. 例外と優先順位

- 軽微な質問・相談・説明依頼のみの場合は Jira を自動起票しない。
- ユーザーから明示的な運用指定がある場合は、その指示を本ルールより優先する。
- 権限/認証エラーは隠さず報告する（必要に応じて `codex mcp login atlassian` を案内）。
